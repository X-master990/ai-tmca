# ADR-001：TMCA 音樂版權授權系統 — 架構決策紀錄

**狀態：** 提案中（Proposed）
**日期：** 2026-05-16
**決策人：** TMCA 理事長／秘書長／承辦 A／承辦 B／會計／核發

---

## 一、Context（現況與動機）

TMCA 目前以一份 `110-115年 總表.xlsx`（4.8 MB、12 張工作表、共約 25,000～50,000 筆紀錄）為核心資料源，4 位內部人員（承辦 A 男 / 承辦 B 女 / 會計 / 核發）共用此 Excel，並用一批 Word 模板套印「證書」、「信封」、「續約通知函」。

### 痛點

1. **重複輸入**：申請書 → 承辦填一次、會計開發票時可能再填一次、核發要再對發票號碼。
2. **多人協作脆弱**：一份 Excel 檔被多人同時編輯，欄位錯誤、版本衝突風險高。
3. **續約難追**：承辦每天忙新案，無法人工逐筆比對「今年到期 vs 同單位是否已自行續約」。
4. **寄信費工**：同名代辦人代辦多家公司、要分掛號/平信、地址常變動，套印模板難維護。
5. **狀態看不到**：核發是否完成、續約是否完成，目前無視覺化燈號。
6. **跨類查找難**：12 張工作表，搜「某代辦人」要逐表查。

### 目標

- 單一資料源（一筆資料貫穿承辦 → 會計 → 核發）
- 「類 Excel」介面（user 明確偏好，最好直接嵌入 Excel 模組）
- 續約自動偵測 + 紅綠燈
- 核發 Word 套印按鈕（證書 + 信封）
- 承辦寄信批次（Word + 信封）
- 全欄位搜尋（含跨表）

### 限制

- 團隊小（4 人）、預算保守、無專職 IT。
- 主要用戶熟悉 Excel，學習新工具的成本要低。
- 中文（繁體）、Taiwan office。
- 資料須留在組織內、不外流。

---

## 二、Decision（推薦結論）

**主推：Microsoft 365 + SharePoint Lists + Power Apps + Power Automate（低代碼方案）**

理由：
1. 介面最接近「類 Excel」需求（Power Apps 表格視圖 ≈ Excel）。
2. SharePoint Lists 是真正的多人共寫資料庫，沒有 Excel 多人編輯衝突問題。
3. Power Automate 原生支援 Word 模板套印（沿用現有 `.docx` 模板，幾乎零改寫）。
4. 月費合理（~NT$300-500/人/月），4 人總額可控。
5. 中文支援完整、政府/公部門/協會普遍採用。
6. 不用養工程師、不用買伺服器。

備案見「四、Options Considered」。

## 三、推薦架構

### 系統分層

```
┌─────────────────────────────────────────────────────────────┐
│  使用者介面層  Power Apps                                    │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐        │
│  │ 承辦 A   │ │ 承辦 B   │ │ 會計     │ │ 核發     │        │
│  │ (單場次) │ │ (其他)   │ │ (發票)   │ │ (證書)   │        │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘        │
│       └────────────┴───────────┴────────────┘                │
│                    │                                          │
│            類 Excel 表格視圖（同一張 List 多重視圖切換）       │
└────────────────────┬─────────────────────────────────────────┘
                     │
┌────────────────────┴─────────────────────────────────────────┐
│  自動化層  Power Automate                                     │
│  • 續約偵測（每日 06:00）                                     │
│  • Word 套印（按鈕觸發）                                      │
│  • 寄信批次（按鈕觸發）                                       │
│  • 狀態燈計算（觸發式）                                       │
└────────────────────┬─────────────────────────────────────────┘
                     │
┌────────────────────┴─────────────────────────────────────────┐
│  資料層  SharePoint Lists                                     │
│  • TotalList（總表，12 個申請類型用「分類」欄區分）           │
│  • TemplateLibrary（Word 模板）                               │
│  • OutputArchive（已產出的證書 / 信封 / 寄信檔）              │
└──────────────────────────────────────────────────────────────┘
```

### 資料模型

**主表：TotalList**（沿用現有 41 欄結構，新增 4 個系統欄位）

| 欄位類別 | 欄位 | 來源 |
|---|---|---|
| 識別 | NO. / 證書編號（**主鍵**）/ 申請類型（**新；12 選 1**） | 系統自動 / 承辦 |
| 證書 | 發證日、註記 | 核發 |
| 發票 | 發票日期、形式、抬頭、統編、發票號碼、金額(含稅) | 會計＋核發 |
| 承辦 | 提報、承辦人、辦理項目（新申辦/續約/授權延長）、申請日期 | 承辦 |
| 申請人 | 申請人、身分證、行動/電話/傳真 | 承辦 |
| 持證者 | 持證者、性質、統一編號（**用於續約比對**） | 承辦 |
| 使用地址 | 郵遞區號、使用地址 | 承辦 |
| 現場 | 現場聯絡人、手機/電話/分機/傳真 | 承辦 |
| 標的 | 台數、廠牌、機號 | 承辦 |
| 授權期間 | 起年/月/日、**迄年/月/日（用於續約比對）** | 承辦 |
| 寄證地址 | **確認(掛號/平信)（新）**、郵區、地址、收件人、收件人電話 | 承辦 |
| **狀態（新）** | 核發狀態（紅/綠）、續約狀態（紅/綠）、最後更新時間 | 系統 |

**附表：**
- `TemplateLibrary`：證書模板（27 種）、信封模板、續約函模板
- `OutputArchive`：已產出的 Word 檔 + 產出時間 + 產出人

### 業務流程對應

| 階段 | 角色 | 系統動作 |
|---|---|---|
| 進件 | 申請人 | （線下）申請書 LINE 給承辦、匯款銀行 |
| 前置 | 會計 | 開匯款單 → 轉交承辦 |
| Stage 1 | 承辦 A/B | 從「待認領清單」依申請類型自挑（單場次→A、其他→B）；登入各自視圖；填承辦欄位；上傳附件 |
| Stage 2 | 會計 | 切換「會計視圖」（自動帶承辦欄位為唯讀）；填發票欄位；連動發票機 |
| Stage 2.5 | 承辦 | 仍可看全欄位、改全欄位（覆蓋會計欄位 OK，留稽核軌跡） |
| Stage 3 | 核發 | 切換「核發視圖」；輸入發票號碼 → 該筆**核發狀態燈轉綠**；點「產生」→ Power Automate 抓總表欄位填模板，產出 證書.docx + 信封.docx |
| 寄信 | 承辦 | 進「續約資料」分類；查「未續約名單」「續約名單」（系統自動分流）；點「產生寄信檔」→ 產出 Word 通知函 + 信封 |

### 自動化邏輯

**續約偵測（每日排程）**
```
For each row in TotalList:
    expiry = (授權期間-迄年月日)
    same_unit_rows = filter(統一編號 == row.統一編號 OR 持證者 == row.持證者)
    has_renewal = any(other.迄日 > row.迄日 for other in same_unit_rows)

    if has_renewal:
        row.續約狀態 = "綠（已續約）"
    elif expiry <= today + 30 days:
        row.續約狀態 = "紅（≤30 天到期）"
    else:
        row.續約狀態 = "綠"
```

**核發狀態（觸發式）**
```
On TotalList row update:
    if 發票號碼 was just set:
        row.核發狀態 = "綠（核發資料 OK）"
```

**Word 套印（按鈕觸發）**
```
On click "產生" by 核發人員:
    1. 依「申請類型」找對應證書模板 (27 種 → 用 lookup 表)
    2. Power Automate「填入 Word 模板的欄位」action 把總表欄位塞進去
    3. 產出 證書.docx + 信封.docx
    4. 存到 OutputArchive；提供下載連結
```

**寄信批次（按鈕觸發）**
```
On click "產生寄信檔" by 承辦:
    1. 拿目前篩選後的清單（未續約名單 / 續約名單）
    2. 依「持證者」群組化（同名代辦人合併）
    3. 用 續約函.docx 模板填入：收件人、地址、確認(掛號/平信)
    4. 產出 通知.docx + 信封.docx（每組一份或合併一份皆可）
```

### 權限模型

| 角色 | 讀 | 寫 |
|---|---|---|
| 承辦 A | 全部 | 全部（含會計欄位，留稽核軌跡） |
| 承辦 B | 全部 | 全部（同上） |
| 會計 | 全部 | 會計欄位 + 標記發票完成 |
| 核發 | 全部 | 核發欄位 + 發票號碼補登 + 觸發產生 |
| 理事長/秘書長 | 全部 | 唯讀 |

## 四、Options Considered

### Option A：Microsoft 365 + Power Platform（**推薦**）

| 維度 | 評估 |
|---|---|
| 複雜度 | 低-中（低代碼） |
| 一次性成本 | NT$50K-150K（建置） |
| 月費 | ~NT$2,000-3,000（4 人 + Power Apps 授權） |
| 擴充性 | 高（可加流程、報表、Power BI 圖表） |
| 團隊熟悉度 | 中（用過 Excel/Word 即可；Power Apps 要學） |
| Word 套印 | ★★★★★（原生支援） |
| 多人協作 | ★★★★★（原生支援） |

**Pros**
- 「類 Excel」介面最像
- Word 模板可以原樣搬過去
- 雲端、無須維運
- 中文／繁體完整

**Cons**
- 月費長期成本
- 嚴格客製要學 Power Fx 公式
- 資料在雲端（協會須確認資料政策）

---

### Option B：客製 Web App（React + Handsontable + Node + PostgreSQL）

| 維度 | 評估 |
|---|---|
| 複雜度 | 高 |
| 一次性成本 | NT$300K-600K |
| 月費 | NT$500-1,500（雲端代管） |
| 擴充性 | 極高 |
| 團隊熟悉度 | 低（要找工程師） |
| Word 套印 | 需自寫（docxtemplater / python-docx） |
| 多人協作 | 高 |

**Pros**
- 100% 客製
- 不被綁特定廠商
- Handsontable / AG-Grid 介面非常像 Excel

**Cons**
- 開發週期 3-6 個月
- 後續要持續維護（找誰修 bug？）
- 4 人團隊用這規模有點 overkill

---

### Option C：強化 Excel + VBA + 共用網路磁碟（最低成本）

| 維度 | 評估 |
|---|---|
| 複雜度 | 低 |
| 一次性成本 | NT$20K-50K（VBA 開發） |
| 月費 | 0 |
| 擴充性 | 低 |
| 團隊熟悉度 | 高 |
| Word 套印 | ★★★（VBA + Word Mail Merge） |
| 多人協作 | ★（Excel 不適合多人即時編輯） |

**Pros**
- 零月費、純地端
- 跟現況 99% 相容

**Cons**
- Excel 多人寫同檔常衝突、檔案損毀風險
- VBA 維護人員難找
- 不能算「系統」，比較像加強版手工流程
- 4.8 MB 檔案再長下去會卡頓

---

### Option D：Microsoft Access 桌面資料庫 + Word Mail Merge

| 維度 | 評估 |
|---|---|
| 複雜度 | 中 |
| 一次性成本 | NT$80K-200K |
| 月費 | 0（內含 Office） |
| 擴充性 | 中 |
| 團隊熟悉度 | 低 |
| Word 套印 | ★★★★（Access 原生 Mail Merge） |
| 多人協作 | 中（區網檔案分享） |

**Pros**
- 沒有月費
- Access ⇄ Word 整合好

**Cons**
- 微軟逐步淡化 Access
- 雲端／遠端難整合
- 介面感覺老氣

## 五、Trade-off Analysis

| 維度 | Option A | Option B | Option C | Option D |
|---|:-:|:-:|:-:|:-:|
| 上線速度 | 6-10 週 | 12-24 週 | 4-6 週 | 8-12 週 |
| 月費 | 中 | 低-中 | 零 | 零 |
| 「像 Excel」 | ★★★★ | ★★★★★ | ★★★★★ | ★★★ |
| 多人協作 | ★★★★★ | ★★★★★ | ★ | ★★★ |
| 維運難度 | 低 | 高 | 中 | 中 |
| 廠商鎖定 | 微軟 | 無 | 無 | 微軟 |
| 規模擴充 | 高 | 極高 | 低 | 中 |

**關鍵取捨**
- 想「最像 Excel」：選 C（純 Excel）或 B（Handsontable）
- 想「最快上線、最穩」：選 A
- 想「零月費」：選 C 或 D
- 想「未來不被綁」：選 B

對 TMCA 的具體建議：**選 A**，因為多人協作 + Word 套印 + 上線速度三者同時兼顧、且月費可負擔。

## 六、Consequences

**會變容易**
- 申請進件後，承辦/會計/核發共用同一筆資料，不再重複輸入。
- 續約名單自動跑出來，承辦不用每天比對。
- 寄信檔一鍵產生，掛號/平信不會弄錯。
- 多人同時改資料不會壞檔。

**會變難**
- 要養成「不要直接編 Excel 檔」的習慣（資料源變成 SharePoint List）。
- 系統壞了要懂 Power Platform 才能修（建議建置時請廠商一併簽 1-2 年維運合約）。

**要再檢視**
- 6 個月後評估月費是否合理、有沒有冗用功能可關。
- 若 SharePoint List 列數逼近 30,000 上限（單表），需要做封存策略。

4. [ ] 把 12 張工作表合併成單一「申請類型」分類欄

### Phase 2：建置主系統（3 週）
1. [ ] 建立 SharePoint List「TotalList」
2. [ ] 把 Excel 資料匯入 List
3. [ ] 在 Power Apps 建 4 個角色視圖（承辦 A / B / 會計 / 核發）
4. [ ] 設定權限：承辦可改全欄位、會計只改會計欄位、核發只改核發欄位

### Phase 3：自動化（3 週）
1. [ ] Power Automate：續約偵測排程
2. [ ] Power Automate：核發 Word 套印（27 種證書模板）
3. [ ] Power Automate：核發信封套印
4. [ ] Power Automate：承辦寄信檔批次（Word 通知 + 信封）

### Phase 4：狀態燈 & 搜尋（1 週）
1. [ ] 核發狀態燈（綠/紅）依「發票號碼是否已填」
2. [ ] 續約狀態燈（綠/紅）依續約偵測結果
3. [ ] 全欄位搜尋／代辦人交叉查詢視圖

### Phase 5：試運行（2-4 週）
1. [ ] 4 人並行使用、舊 Excel 同步維護

### Phase 6：上線後維運（持續）
1. [ ] 每月匯出備份（雲端 → 地端 Excel）
2. [ ] 每年檢視欄位、模板是否需更新
3. [ ] 規劃 3 年後資料封存策略

**估算總工期：11-15 週（含試運行）**
**估算總成本：NT$100K-200K（建置）+ NT$30K-40K/年（月費 + 維運）**

## 九、未採納原因記錄

- **不選 Option B（自製 Web）**：團隊 4 人、無工程師，後續維運找不到人。等規模到 10 人以上、或需要對外開放，再考慮重寫。
- **不選 Option C（Excel + VBA）**：多人協作風險已是當前最大痛點，繼續用 Excel 等於沒解決問題。
- **不選 Option D（Access）**：微軟戰略性淡化，再投入是逆風。
